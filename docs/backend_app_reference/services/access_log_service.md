# access_log_service.py

## Для чего этот файл

Этот сервис отвечает не за распознавание, а за красивую подачу событий прохода. Когда камера уже решила “доступ разрешён” или “неизвестное лицо”, в базе появляется `AccessLog`. Этот файл берёт такую запись, подтягивает к ней сотрудника/гостя/камеру и превращает в понятный объект для frontend.

Пример: в базе хранится `guest_id=7`, `camera_id=3`, `status=granted`. Frontend не должен сам собирать имя гостя и название камеры. Сервис отдаёт уже готовый payload: “`[Гость] Иванов Иван`, `UKSIVT Camera02`, `granted`, время”.

## Как работает по шагам

1. Берётся запись `AccessLog`.
2. Через join подтягивается `Employee`, `Guest` и `Camera`.
3. `build_person_name` решает, как подписать человека.
4. `build_access_log_payload` собирает короткий JSON для UI.
5. `publish_access_log_created` отправляет событие в WebSocket topic `access_logs`.

## Главные функции

| Функция | Простое объяснение |
|---|---|
| `build_person_name` | Если есть сотрудник, возвращает его ФИО. Если есть гость, добавляет префикс `[Гость]`. Если никого нет — пишет `Неизвестное лицо`. |
| `build_access_log_payload` | Делает из записи БД объект, который удобно показать в журнале событий. |
| `get_recent_access_logs` | Берёт последние события журнала, например для первичной загрузки страницы. |
| `get_access_log_payload` | Получает один конкретный лог по id и собирает payload. |
| `publish_access_log_created` | После создания нового `AccessLog` отправляет событие frontend через WebSocket. |

## Где используется

Главный вызов идёт из `stream_manager.py`. Когда `_handle_access` распознал человека и записал `AccessLog`, он вызывает `publish_access_log_created`, чтобы интерфейс сразу увидел новое событие.

## Важно понимать

Если в UI написано “Неизвестное лицо”, этот файл не “плохо распознал”. Он только показывает то, что ему дали. Распознавание происходит раньше, в `recognition_service.py` и `stream_manager.py`.
