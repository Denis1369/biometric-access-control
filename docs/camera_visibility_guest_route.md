# Зоны видимости камер и вероятный маршрут гостя

## 1. Назначение модуля

Модуль связывает события распознавания гостя на камерах с планом этажа. Для этого в системе используются три слоя данных:

- граф возможных маршрутов по этажу;
- зоны видимости камер;
- события распознавания гостя из `TrackingLog` и, при необходимости, `AccessLog`.

Результатом является вероятный маршрут перемещения гостя по зданию, отображаемый поверх плана этажа.

Формулировка для диплома:

> В системе реализован модуль построения вероятного маршрута перемещения гостя по плану здания. Для этого оператор задаёт граф возможных маршрутов и зоны видимости камер. При фиксации гостя камерой система определяет участок графа, попадающий в область наблюдения данной камеры, а затем по последовательности событий распознавания строит наиболее вероятный маршрут перемещения с использованием алгоритма поиска кратчайшего пути.

## 2. Почему зона видимости задаётся 4-угольником

Камеры в помещении часто установлены под углом, поэтому их область наблюдения на плане удобнее задавать не окружностью, а произвольным четырёхугольником. Такой формат позволяет быстро приблизительно отметить сектор камеры: часть коридора, входную зону, поворот или участок холла.

Зона задаётся вручную, потому что автоматическое вычисление поля зрения требует точной калибровки камеры, высоты установки, фокусного расстояния, угла наклона и геометрии помещения. Для дипломного проекта и практической настройки операторская разметка быстрее, прозрачнее и надёжнее.

## 3. Почему маршрут вероятный

Система не знает точную координату человека внутри кадра на плане здания. Если гость распознан камерой, считается только, что он находился где-то внутри зоны видимости этой камеры.

Поэтому маршрут не является абсолютной траекторией. Это наиболее вероятный путь по заранее размеченному графу между зонами камер, где гость был зафиксирован последовательно по времени.

## 4. Как оператор размечает граф маршрутов

Оператор открывает план этажа и в режиме разметки ставит технические точки графа. Затем соединяет точки линиями. Точки не имеют названий и типов: это только вершины графа с координатами `x` и `y` в пикселях оригинального изображения плана.

Линии графа описывают проходы, по которым реально может двигаться человек.

## 5. Как оператор задаёт зону видимости камеры

Оператор нажимает `Редактировать камеры`, выбирает камеру на плане и в панели настройки камеры нажимает `Задать зону`. После этого он кликает четыре точки на плане. После четвёртой точки появляется полупрозрачный четырёхугольник.

Вершины зоны можно перетаскивать. Зону можно сохранить, сбросить или удалить. Координаты зоны сохраняются в пикселях оригинального изображения плана.

Пример зоны:

```json
{
  "floor_id": 1,
  "points": [
    { "x": 120.0, "y": 240.0 },
    { "x": 320.0, "y": 210.0 },
    { "x": 360.0, "y": 420.0 },
    { "x": 100.0, "y": 450.0 }
  ]
}
```

## 6. Как события связываются с камерами

Основной источник событий перемещения гостя — таблица `tracking_logs`. Каждая запись содержит `guest_id`, `camera_id`, `timestamp` и confidence распознавания.

Если в `TrackingLog` за выбранный период нет событий, система может использовать `AccessLog` как fallback, если в нём есть `guest_id`, `camera_id` и `timestamp`.

События фильтруются по гостю, этажу и временному интервалу, затем сортируются по `timestamp`.

## 7. Как зона камеры пересекается с линиями маршрута

Для каждой камеры система берёт её четырёхугольник видимости и проверяет все линии графа маршрутов этажа. Линия считается попавшей в зону камеры, если выполняется хотя бы одно условие:

- один из концов линии находится внутри зоны;
- линия пересекает любую сторону четырёхугольника.

Проверка реализована без тяжёлых геометрических библиотек.

## 8. Как выбираются candidate_node_ids

Если линия графа пересекает зону камеры, обе её вершины считаются возможными точками маршрута для этой камеры. Эти вершины попадают в `candidate_node_ids`.

Если зона камеры не пересекается с графом, система добавляет warning. Это сигнал оператору, что нужно поправить граф или зону видимости камеры.

## 9. Как строится путь между событиями камер

После сортировки событий система рассматривает пары соседних камер: `A -> B`, `B -> C` и так далее.

Для каждой камеры берутся её `candidate_node_ids`. Если у камеры несколько возможных точек, система перебирает пары candidate-узлов между соседними камерами и выбирает путь с минимальной длиной. Для поиска пути используется алгоритм Дейкстры, уже применяемый в графе маршрутов.

Если путь между двумя зонами не найден, весь ответ не падает: система добавляет предупреждение и продолжает обработку остальных участков.

## 10. Как показывается направление движения

Направление движения определяется только порядком событий по `timestamp`. Система не пытается вычислять направление из видео.

На frontend события камер отображаются номерами `1`, `2`, `3`, а вероятный маршрут рисуется линией со стрелками. Номер события показывает порядок фиксации гостя камерами.

## 11. Применение в биометрическом контроле доступа

После регистрации гостя система может фиксировать его появление на разных камерах с помощью распознавания лица или Re-ID. Зоны камер связывают эти события с участками маршрута на плане этажа. В результате оператор видит не просто список камер, а понятное вероятное перемещение гостя внутри здания.

Такой подход подходит для дипломного проекта, потому что он разделяет ответственность:

- биометрический модуль отвечает за распознавание гостя;
- камеры отвечают за события наблюдения;
- ручной граф отвечает за допустимые маршруты;
- зоны видимости связывают камеры с участками графа.
